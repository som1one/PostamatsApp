# Plan: separate "Inventory placement" section + simplified "Lockers" section

## Goal

Simplify the Postamats admin so that a non-technical operator can:
- Manage lockers via a clean section that focuses on the locker itself, not inventory wiring.
- Use a **new dedicated section "Размещение товаров"** (Inventory placement) where they:
  - Pick a locker, pick a free cell, pick a product from the catalog, click "Положить в ячейку" → backend records that this product is in this cell, the unit becomes `available` for renting, and it shows up in the public catalog.
  - In the same place, "Забрать на обслуживание" → backend opens the cell, removes the unit from the cell, marks it `MAINTENANCE` (with optional comment), and an `InventoryMovement` row is written for audit.

User answers:
- Simplification level: maximally simple, designed for non-technical operators.
- Inventory source: use existing `InventoryUnit` rows when available; otherwise auto-create one for the chosen product (no SN required).
- Maintenance flow: one-click "забрать на обслуживание" (no two-stage workflow).

## Current architecture (what exists today)

Backend models:
- `LockerLocation` (id, city_id, name, address, status, external_locker_id, …) – `backend/models/locker_location.py`.
- `LockerCell` (id, locker_id, label, external_cell_id, size, status `vacant|occupied|reserved|opened|fault|disabled`, supports_return) – `backend/models/locker_cell.py`.
- `InventoryUnit` (id, product_id, locker_cell_id (unique nullable), serial_number nullable, status `available|reserved|rented|return_pending|damaged|maintenance|lost|retired`) – `backend/models/inventory_unit.py:12`.
- `InventoryMovement` (audit trail of unit moves) – `backend/models/inventory_movement.py:10`.
- `Product`, `ProductCategory`.

Backend admin endpoints used today (`backend/routers/admin/lockers.py`):
- `POST /api/admin/lockers/{locker_id}/cells/{cell_id}/assign-unit` – attaches an existing `InventoryUnit` (passed by UUID) to a cell, calls `sync_cell_state` (ESI), writes audit + InventoryMovement. `lockers.py:447`.
- `DELETE /api/admin/lockers/{locker_id}/cells/{cell_id}/assignment` – detaches a unit, syncs ESI to `vacant`. `lockers.py:535`.
- `POST /api/admin/lockers/{locker_id}/open-cell` – ESI open. `lockers.py:308`.
- `GET /api/admin/lockers/{locker_id}` – returns locker, cells (each with its inventoryUnit), `assignableUnits` (free units in DB), incidents, recent rental events. `lockers.py:599`.
- `POST /api/admin/lockers/{locker_id}/cells` and `PATCH /…/cells/{cell_id}` – cell CRUD.

Admin UI (`admin/index.html`, `admin/app.js`):
- "Постаматы" section opens `locker-detail-modal` rendered by `renderLockerDetailModal()` (`app.js:1807`). It includes:
  - locker editor (name/address/city/status/partnerName/externalLockerId/externalProvider/lat/lon/workingHours JSON),
  - cells table with per-row "Привязать"/"Снять"/"Сохранить"/"Открыть",
  - inline "UUID юнита" input + a `<datalist>` of `assignableUnits`,
  - "Доступные товары" summary, incidents, rental events.

Reality: the current "Привязать" UX requires the operator to type or pick a UUID. There is no UI to *create* an `InventoryUnit` from a product – units only appear via `backend/scripts/create_test_data.py`. This is the root cause the user is asking to fix.

## What we change

### 1. Backend: new admin endpoints for inventory placement

Add a new router `backend/routers/admin/inventory.py` mounted at `/api/admin/inventory` (registered in `backend/main.py`). It owns the operator-friendly placement flow:

- `GET /api/admin/inventory/lockers` – lightweight list `[{id, name, cityName, address, freeCellCount, occupiedCellCount}]` for the placement screen's locker dropdown.
- `GET /api/admin/inventory/lockers/{locker_id}/cells` – list of cells of a locker with their current placement: `{id, label, externalCellId, size, status, supportsReturn, currentUnit?: {id, productId, productName, status, serialNumber}}`. Used to pick a free cell.
- `GET /api/admin/inventory/products?search=&active=` – simple paginated catalog list `[{id, name, slug, categoryName, isActive, totalAvailableUnits}]`.
- `POST /api/admin/inventory/cells/{cell_id}/place` – body `{productId: UUID, comment?: string}`. Server-side logic:
  1. Validate cell status is `vacant`/`disabled`-with-no-unit and not currently occupied.
  2. Find a free `InventoryUnit` of `productId` with `status=AVAILABLE` and `locker_cell_id=NULL`. If none exists, create a new `InventoryUnit(product_id=productId, status=AVAILABLE)`.
  3. Set `unit.locker_cell_id = cell_id`, `cell.status = OCCUPIED`.
  4. Best-effort `sync_cell_state(state="occupied")` (ESI). On `EsiOpenError` → 502 with detail; if ESI is not configured, the existing client returns gracefully so DB state still applies.
  5. `add_inventory_movement(reason="admin_place_in_cell", comment=...)`.
  6. `record_admin_audit("inventory.place")`.
  7. Returns the updated cell row.
- `POST /api/admin/inventory/cells/{cell_id}/take-for-service` – body `{reason?: string, openCell?: bool = true, targetStatus?: "maintenance"|"damaged" = "maintenance"}`:
  1. Look up the unit currently in this cell.
  2. If `openCell` is true, send ESI `admin_trigger_open_cell` (same call existing `admin_open_cell` uses, `lockers.py:308`). Failures here are reported but do NOT prevent the DB state change unless `openCell=true` and ESI is configured – clear toast on partial success vs failure.
  3. `unit.locker_cell_id = NULL`, `unit.status = MAINTENANCE` (or `DAMAGED`).
  4. `cell.status = VACANT`.
  5. `sync_cell_state(state="vacant")` best effort.
  6. `add_inventory_movement(reason="admin_take_for_service", comment=reason)`.
  7. `record_admin_audit("inventory.take_for_service", payload={cellId, unitId, targetStatus})`.
  8. Returns the unit + cell after the change.

Reuse helpers (no new utils needed):
- `backend/utils/inventory_tracking.add_inventory_movement` (`inventory_tracking.py:10`)
- `backend/utils/esi_client.sync_cell_state` and `admin_trigger_open_cell`.
- `backend/utils/admin_audit.record_admin_audit`.

Schemas in `backend/schemas/admin_panel_schemas.py`:
- `AdminPlaceProductInCellPayload { productId: UUID, comment?: str }`.
- `AdminTakeForServicePayload { reason?: str, openCell?: bool = True, targetStatus?: Literal["maintenance","damaged"] = "maintenance" }`.

DB migrations: not required. Existing `InventoryUnit.status` already includes `MAINTENANCE` and `DAMAGED`. `InventoryMovement` already supports the new `reason` strings.

### 6. Tests

Backend unit tests under `backend/tests/test_admin_inventory_placement.py`:
- placement on a vacant cell with no existing free units → creates a new unit, status=available, attached.
- placement on a vacant cell with an existing free unit of that product → reuses it (no new unit created), status=available, attached.
- placement on an already-occupied cell → 409 `CELL_ALREADY_OCCUPIED`.
- take-for-service from an occupied cell → unit detached, unit.status=maintenance, cell.status=vacant, InventoryMovement created with `reason=admin_take_for_service`.
- take-for-service with `targetStatus=damaged` → status=damaged.
- take-for-service from an empty cell → 404 `CELL_HAS_NO_UNIT`.
- ensures admin audit row is recorded for both endpoints.

Run `pytest backend/tests/test_admin_inventory_placement.py -q` plus `pytest backend/tests` to confirm no regressions in existing locker/cell tests.

## Risks and mitigations

- **ESI errors during placement**: the existing `sync_cell_state` raises `EsiOpenError` only on real failures. We follow the existing pattern in `admin_assign_inventory_unit` (`lockers.py:482`) and roll back DB state on ESI errors so the state remains consistent.
- **Race condition on cell occupancy**: wrap placement in a `SELECT … FOR UPDATE` (or explicit re-check inside the same transaction) to avoid two operators placing into the same cell simultaneously.
- **Public catalog visibility**: the catalog already uses `InventoryUnit.status == AVAILABLE` and `locker_cell_id IS NOT NULL` (see `backend/utils/lockers_utils.aggregate_available_inventory_by_product`), so the placement immediately surfaces the product in apps. No frontend changes needed there.
- **Backwards compatibility**: legacy endpoints `/assign-unit` and `/assignment` and the existing `assignableUnits` payload field are kept; only the UI stops using them. Tests for them remain intact.
- **Audit volume**: each placement/service action writes one `admin_audit_event` and one `inventory_movement` – consistent with existing flow.

## Out of scope

- Bulk placement (multiple products in one go). Can be added later.
- Returning a unit from maintenance back into the catalog – out of this iteration; a separate small "Maintenance shelf" view can come next if needed.
- Locker-level inventory reports / export.
