# MVP на 15 дней: фронтенд и бэкенд

Этот документ фиксирует минимальный MVP сервиса аренды товаров через постаматы.
Он написан на русском и нужен как общая рабочая рамка для двух направлений:

- `backend`: что обязательно должно быть реализовано на сервере
- `frontend`: что должно быть в мобильном приложении и админке

Важно:

- я выполняю только `frontend`
- backend реализуется отдельно
- backend API подробно описан в файле [api-mvp-contract.md](/Users/cultim/Documents/postamats/docs/api-mvp-contract.md)

## 1. Цель MVP

Запустить пилотную версию, в которой пользователь может:

1. войти по телефону;
2. пройти ручную верификацию;
3. выбрать город и постамат;
4. найти товар;
5. забронировать его;
6. пройти предавторизацию оплаты;
7. получить товар через постамат;
8. вернуть товар;
9. посмотреть историю аренд.

Это не финальный продукт, а рабочий pilot.

## 2. Что входит в MVP

### Обязательно

- вход по телефону и SMS
- профиль пользователя
- ручная верификация через админку
- выбор города
- карта постаматов
- каталог товаров
- карточка товара
- расчёт цены
- бронирование
- предавторизация через ЮKassa
- старт аренды после открытия ячейки
- возврат товара
- история аренд
- базовая админка для операторов

### Не входит в первую поставку

- полноценная автоматическая KYC-проверка
- роли франчайзи
- расширенная аналитика
- выгрузки Excel/PDF
- почасовая аренда
- автопродление просрочки
- сложный сценарий инцидентов
- богатая система лояльности и промокоды

## 3. Главный пользовательский сценарий

1. Пользователь открывает приложение.
2. Входит по номеру телефона.
3. Загружает данные для верификации.
4. После подтверждения выбирает город.
5. Видит карту постаматов и каталог.
6. Заходит в карточку товара.
7. Выбирает постамат и срок аренды.
8. Видит итоговую стоимость.
9. Подтверждает бронь и проходит предавторизацию.
10. Получает PIN для выдачи.
11. Подходит к постамату и забирает товар.
12. Во время аренды видит активный заказ.
13. Инициирует возврат.
14. Возвращает товар в ячейку.
15. Видит завершённую аренду в истории.

## 4. MVP по backend

Ниже не код, а обязательные серверные зоны ответственности.

### 4.1 Авторизация и сессия

Нужно реализовать:

- вход по телефону
- подтверждение SMS-кода
- refresh token
- получение текущего пользователя

Зачем:

- это входная точка во всё приложение
- без статуса пользователя нельзя правильно открывать аренду, карту и каталог

### 4.2 Верификация пользователя

Нужно реализовать:

- приём данных пользователя
- загрузку фото документов
- хранение статуса проверки
- ручное подтверждение или отклонение из админки

Зачем:

- аренда без залога требует доверия к пользователю
- это бизнес-блокер для старта аренды

### 4.3 Города и постаматы

Нужно реализовать:

- список городов
- список постаматов по городу
- статусы постаматов
- доступность товаров в конкретной точке

Зачем:

- карта и выбор точки являются ядром продукта

### 4.4 Каталог и доступность

Нужно реализовать:

- категории
- список товаров
- карточку товара
- характеристики
- фотографии
- наличие по постаматам
- расчёт тарифов

Зачем:

- пользователь должен быстро понять, что можно взять прямо сейчас и где

### 4.5 Бронирование

Нужно реализовать:

- расчёт quote перед бронированием
- создание брони
- время жизни брони
- отмену брони

Зачем:

- без этого нельзя безопасно перейти к оплате и удержанию конкретного товара

### 4.6 Платежи

Нужно реализовать:

- предавторизацию в ЮKassa
- webhook от ЮKassa
- проверку статуса оплаты
- привязку оплаты к брони и аренде

Зачем:

- деньги и аренда должны жить в одной бизнес-цепочке

### 4.7 Аренда

Нужно реализовать:

- подтверждение брони после оплаты
- генерацию PIN
- перевод аренды в статус выдачи
- перевод аренды в активное состояние
- возврат
- завершение аренды
- историю аренд

Зачем:

- это главный доменный поток продукта

### 4.8 Интеграция с постаматами

Нужно реализовать:

- резерв ячейки
- открытие ячейки
- обработку webhook-событий
- статусы online/offline
- повторные попытки при ошибке

Зачем:

- без надёжной синхронизации с постаматом аренда будет ломаться в бою

### 4.9 Админка

Нужно реализовать:

- вход администратора
- просмотр пользователей
- ручное подтверждение верификации
- просмотр постаматов
- просмотр аренд
- просмотр товаров и остатков
- ручные операторские действия при сбое

Зачем:

- для MVP часть сложных кейсов будет закрываться оператором вручную

Нужные экраны:

- экран бронирования
- экран оплаты
- экран успешной брони
- экран активной аренды
- экран возврата
- история аренд
- профиль
- экран поддержки / ошибки

### 5.2 Что должно быть на фронте обязательно

- понятная навигация между основными экранами
- работа с авторизацией и токенами
- отображение статусов пользователя
- карта с точками
- каталог с карточками
- фильтры базового уровня
- экран checkout
- экран активной аренды
- экран истории
- загрузка документов на верификацию
- состояния загрузки, пустых данных и ошибок

### 5.3 Что можно упростить на фронте

- анимации свести к минимуму
- сложные фильтры не делать
- deep link не делать
- богатый чат поддержки не делать
- промо-блоки и маркетинг свести к минимуму

## 6. Минимальная админка во frontend

Если админка тоже идёт во фронтенд-контур текущего этапа, то минимум такой:

- логин
- список пользователей
- карточка пользователя
- список аренд
- карточка аренды
- список постаматов
- карточка постамата
- список товаров
- список физических единиц товара

Для MVP админка должна быть в первую очередь операционной, а не красивой.

## 7. Стиль интерфейса для MVP

Ориентир по референсу:

- чистый светлый интерфейс
- крупные карточки
- один яркий брендовый акцентный цвет
- много воздуха
- крупные CTA-кнопки
- минимум визуального шума
- упор на понятные статусы и короткие действия

Принцип:

- приложение должно быть похоже не на маркетплейс, а на простой инструмент получения вещи рядом с домом

## 10. Что нужно заморозить до начала активной реализации

Нужно принять решения по этим пунктам:

1. Можно ли бронировать без завершённой верификации?
2. Возврат только в свой постамат или в любой?
3. Когда списываются деньги: в начале аренды или в конце?
4. Что делать, если оплата прошла, а ячейка не зарезервировалась?
5. Сколько живёт бронь?
6. Что происходит при просрочке в MVP?
7. Как оператор завершает аренду вручную, если webhook не пришёл?
