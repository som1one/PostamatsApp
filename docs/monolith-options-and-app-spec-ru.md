# Варианты монолита и спецификация приложения

Документ нужен, чтобы быстро выбрать архитектуру и зафиксировать, что именно
мы строим в MVP и во что это должно вырасти дальше.

## 1. Что строим

Приложение для аренды товаров через постаматы.

Пользовательский сценарий:

1. пользователь входит по телефону;
2. проходит верификацию;
3. выбирает город и постамат;
4. открывает каталог;
5. выбирает товар и срок аренды;
6. проходит предавторизацию оплаты;
7. получает PIN для выдачи;
8. забирает товар в постамате;
9. возвращает товар;
10. видит историю аренд.

Система состоит из:

- мобильного приложения для клиента;
- web-админки для операторов;
- backend-монолита;
- интеграций с постаматами, платежами и push.

## 2. Несколько вариантов монолита

Ниже не абстрактные варианты, а реальные схемы, которые подойдут под ваш MVP.

### Вариант A. TypeScript modular monolith

Состав:

- `Backend`: NestJS или Fastify + TypeScript
- `DB`: PostgreSQL
- `Cache/queues`: Redis
- `Admin frontend`: React + Vite
- `Mobile`: Expo / React Native
- `Contracts`: Zod / OpenAPI / shared types

Как устроен монолит:

- один backend-репозиторий;
- один deploy backend-приложения;
- внутри backend код разбит на модули:
  - `auth`
  - `verification`
  - `cities`
  - `lockers`
  - `catalog`
  - `inventory`
  - `reservations`
  - `payments`
  - `rentals`
  - `notifications`
  - `admin`
  - `integrations.esi`
  - `integrations.yookassa`

Плюсы:

- единый язык на фронте и бэке;
- проще шарить типы и DTO;
- удобно держать одну модель данных;
- легче поддерживать одному сильному fullstack-разработчику;
- хороший вариант, если вы хотите быстро подключать frontend к стабилизированным контрактам.

Минусы:

- если backend пишется без дисциплины, Nest/Fastify быстро превращается в "всё во всём";
- много инфраструктурных решений придётся принимать руками;
- админка не идёт "из коробки", её надо отдельно собирать.

Когда выбирать:

- если команда сильнее в JavaScript/TypeScript;
- если хочется максимальную согласованность с frontend;
- если backend потом может быть постепенно разделён на сервисы без смены языка.

### Вариант B. Django modular monolith

Состав:

- `Backend`: Django + Django REST Framework
- `DB`: PostgreSQL
- `Cache/async`: Redis + Celery или Django-Q
- `Admin`: Django Admin или Django Admin + кастомные экраны
- `Mobile`: Expo / React Native

Как устроен монолит:

- один Django-проект;
- приложения внутри:
  - `users`
  - `verification`
  - `cities`
  - `lockers`
  - `catalog`
  - `inventory`
  - `reservations`
  - `payments`
  - `rentals`
  - `support`
  - `notifications`
  - `integrations`
  - `ops`

Плюсы:

- самый быстрый путь до рабочей операторской админки;
- удобные модели, permissions, forms, admin actions;
- отлично подходит для бизнес-продукта с кучей статусов и ручных операций;
- хороший выбор для MVP, где операторские сценарии важнее идеальной инженерной красоты.

Минусы:

- слабее связка с frontend по типам;
- мобильная команда не получает shared DTO автоматически;
- если backend дальше захочет быть очень event-driven, Django потребует аккуратной дисциплины.

Когда выбирать:

- если цель именно быстро запустить pilot;
- если у вас много ручной модерации, операторских действий и административных экранов;
- если backend-команда чувствует себя в Python лучше, чем в Node.

### Вариант C. FastAPI modular monolith

Состав:

- `Backend`: FastAPI
- `DB`: PostgreSQL
- `Cache/async`: Redis + ARQ/Celery/RQ
- `Admin`: отдельная React-админка
- `Mobile`: Expo / React Native

Как устроен монолит:

- один backend deploy;
- роутеры и доменные модули по бизнес-областям;
- pydantic-схемы для API;
- фоновые задачи под webhook, retry, push, reconciliation.

Плюсы:

- быстрый API-слой;
- чистые схемы и документация;
- удобно для интеграций и чётких API-контрактов;
- хорошо подходит, если backend сильнее в Python и хочет лёгкий каркас без тяжёлого Django.

Минусы:

- админку нужно делать отдельно;
- много операционных вещей придётся собирать вручную;
- для сложного backoffice Django обычно выигрывает по скорости.

Когда выбирать:

- если нужен гибкий Python backend;
- если админка не должна быть слишком богатой на старте;
- если команда любит явные схемы и простые модули.

## 3. Что я рекомендую

### Если цель: реально запустить MVP быстро

Я бы рекомендовал `Django modular monolith`.

Почему:

- у вас админка и операторские процессы критичнее, чем "идеальная microservice-ready архитектура";
- будет много ручных действий: модерация верификации, разбор ошибок постамата, ручное завершение аренды, просмотр платежей;
- для pilot важнее быстро получить устойчивую операционную панель, чем красивую инженерную абстракцию.

### Если цель: упростить жизнь frontend и жить в одном типизированном мире

Я бы рекомендовал `TypeScript modular monolith`.

Почему:

- проще синхронизировать API и фронт;
- легче строить общий контракт;
- проще, если основной контур продукта в JS/TS.

### Мой практический выбор под ваш кейс

Если вы пишете backend сами и хотите быстрее запуститься, я бы выбирал так:

1. `Django modular monolith`, если приоритет — скорость запуска и ops.
2. `TypeScript modular monolith`, если приоритет — единый стек и типы.
3. `FastAPI modular monolith`, если любите Python, но не хотите полный Django.

## 6. Роли в системе

### Клиент

Может:

- зарегистрироваться;
- пройти верификацию;
- смотреть постаматы;
- смотреть товары;
- бронировать;
- оплачивать;
- получать и возвращать товар;
- смотреть историю;
- отправлять обращение в поддержку.

### Оператор

Может:

- просматривать пользователей;
- подтверждать или отклонять верификацию;
- смотреть постаматы и их статусы;
- смотреть аренды;
- вручную завершать или отменять проблемные аренды;
- открывать ячейку в аварийном режиме;
- фиксировать инциденты.

### Администратор

Может всё, что оператор, плюс:

- управлять городами;
- управлять каталогом;
- управлять физическими единицами товаров;
- управлять постаматами;
- смотреть базовую аналитику.

### Франчайзи

Не нужен в первом MVP как полноценная роль.
Лучше добавить после запуска pilot.

## 8. Основные функциональные модули

### 8.1 Авторизация

Функции:

- вход по номеру телефона;
- подтверждение SMS-кода;
- refresh token;
- logout;
- получение текущего пользователя.

Ограничения:

- незалогиненный пользователь не должен видеть flow аренды как завершённый путь.

### 8.2 Верификация

Функции:

- ввод паспортных данных;
- загрузка фото;
- отправка на проверку;
- просмотр статуса;
- операторское подтверждение или отклонение.

Статусы:

- `не начата`
- `на проверке`
- `подтверждена`
- `отклонена`
- `заблокирована`

### 8.3 Города и постаматы

Функции:

- выбор города;
- список точек;
- карта точек;
- просмотр статуса постамата;
- просмотр доступных товаров на точке.

Статусы постамата:

- `онлайн`
- `офлайн`
- `обслуживание`
- `деградирован`

### 8.4 Каталог

Функции:

- список категорий;
- каталог товаров;
- карточка товара;
- фильтрация;
- поиск;
- отображение доступности.

Минимум для карточки товара:

- название;
- обложка;
- короткое описание;
- характеристики;
- доступность;
- цена "от";
- правила аренды;
- комплект.

### 8.5 Бронирование

Функции:

- выбор постамата;
- выбор срока;
- расчёт стоимости;
- создание брони;
- таймер жизни брони;
- отмена брони.

Статусы брони:

- `создана`
- `ожидает оплаты`
- `оплачена`
- `подтверждена`
- `истекла`
- `отменена`

### 8.6 Платежи

Функции:

- предавторизация;
- получение статуса оплаты;
- обработка webhook от платёжного провайдера;
- привязка оплаты к аренде.

MVP-подход:

- сначала только один основной сценарий оплаты;
- спорные кейсы оператор обрабатывает вручную.

### 8.7 Аренда

Функции:

- перевод брони в аренду;
- выдача PIN;
- старт аренды после фактического открытия ячейки;
- экран активной аренды;
- возврат;
- завершение аренды;
- история аренд.

Статусы аренды:

- `готова к получению`
- `ячейка открыта`
- `активна`
- `идёт возврат`
- `завершена`
- `просрочена`
- `инцидент`
- `отменена`

### 8.8 Интеграция с постаматами

Функции:

- резерв ячейки;
- установка PIN;
- открытие ячейки;
- обработка событий открытия и закрытия;
- статус online/offline;
- retry при ошибках.

Важно:

- мобильное приложение не должно самостоятельно решать, что аренда началась или завершилась;
- истина должна приходить из backend после событий постамата.

### 8.9 Поддержка и инциденты

Функции:

- сообщение о проблеме;
- типизация инцидента;
- вложения;
- просмотр в админке;
- ручное вмешательство оператора.

Минимум типов инцидентов:

- ячейка не открылась;
- ячейка не закрылась;
- повреждённый товар;
- неполная комплектация;
- ошибка оплаты;
- постамат офлайн.

## 13. Что входит в MVP, а что нет

### Входит

- клиентское мобильное приложение;
- базовая админка;
- ручная верификация;
- одна модель оплаты;
- одна базовая модель аренды;
- интеграция с постаматом по happy path;
- история аренд;
- базовые инциденты.

### Не входит

- полноценная франчайзинговая модель;
- сложная BI-аналитика;
- глубокая автоматизация инцидентов;
- сложные тарифные сетки;
- многошаговая программа лояльности;
- многоуровневые права на уровне регионов и ролей.

## 15. Финальная рекомендация

Если смотреть на задачу трезво, лучший путь такой:

1. backend как `modular monolith`;
2. frontend отдельно, но строго от контрактов;
3. state machine аренды и операторские сценарии — это центр системы;
4. MVP строится вокруг надёжности, а не вокруг количества экранов.

Если бы я выбирал один вариант под ваш текущий кейс, я бы предложил:

- `Django modular monolith` для backend;
- `Expo + React Native` для mobile frontend;
- `React + Vite` для admin frontend.
